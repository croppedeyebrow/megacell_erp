# MegaCell ERP 2.0 제품 기획서

문서 버전: v0.1  
작성일: 2026-07-11  
대상 저장소: `croppedeyebrow/megacell_erp`  
목표 기술 스택: FastAPI + React(TypeScript) + PostgreSQL

---

## 1. 문서 목적

현재 Streamlit 기반 ERP MVP를 업무의 기준 시스템으로 발전시키기 위한 제품 범위, 사용자 경험, 데이터 구조, 기술 방향, 전환 전략을 정의한다.

이 문서는 다음 질문에 답하는 것을 목표로 한다.

- 어떤 문제를 해결할 것인가?
- 누가 어떤 업무를 시스템에서 처리하는가?
- 1차 출시에서 무엇을 만들고 무엇을 미룰 것인가?
- 기존 Streamlit·SQLite·Excel 자산을 어떻게 이전할 것인가?
- 개발 완료와 운영 안정화를 무엇으로 판단할 것인가?

## 2. 프로젝트 배경

### 2.1 현재 상태

현재 프로젝트는 Streamlit, Python, SQLite 기반의 사내 ERP MVP다. 기존 Excel 원장을 SQLite에 적재해 조회하고, 일부 운영 데이터를 별도 테이블에 저장하는 구조다.

저장소 기준으로 확인된 구현 범위는 다음과 같다.

- 자체 로그인, PBKDF2 비밀번호 해시, 사용자·부서·역할 관리
- 부서별 메뉴와 접근 제어
- 수주·미출고 조회
- 제품 재고와 입출고 기록 조회
- 자재 재고, 부품 리스트, BOM, 자재 발주 조회
- 배터리 현재고, 입출고, 미출고 달력, 사용 이력
- AS 이력 조회
- 수주·구매·재고·AS 기반 대시보드와 경영 요약
- Excel 원장 데이터의 SQLite 적재
- PDF 견적서·발주서 분석 및 주문서 XLSM 생성
- ERP 운영 수주, 변경 로그, 재고 이동, 문서, 동기화 상태용 초기 스키마

일부 메뉴는 화면 이름과 설명만 존재하며 실제 업무 처리는 아직 구현되지 않았다. 생산 진행, 출고 요청, 고객 관리, 입찰, 정산·청구, 제품 사양 변경 이력 등이 이에 해당한다.

### 2.2 전환이 필요한 이유

Streamlit MVP는 빠른 검증에는 적합하지만 본격적인 다중 사용자 업무 시스템으로 확장할 때 다음 제약이 있다.

- 화면 상태와 업무 로직의 결합도가 높아 기능 확장이 어렵다.
- 세밀한 권한, 승인 흐름, 알림, 실시간 상태 반영 구현이 제한된다.
- API가 없어 외부 서비스 및 자동화 연동이 어렵다.
- SQLite와 파일 기반 동기화는 동시 쓰기와 데이터 정합성 관리에 한계가 있다.
- 변경 이력, 감사 로그, 장애 복구, 배포 자동화 체계를 강화할 필요가 있다.

## 3. 제품 비전과 목표

### 3.1 제품 비전

> 수주부터 구매·생산·재고·출고·AS까지 하나의 업무 흐름으로 연결하고, 모든 변경의 담당자와 이력을 추적할 수 있는 메가셀의 사내 운영 플랫폼

### 3.2 1차 목표

1. 기존 ERP의 핵심 조회 기능을 중단 없이 이전한다.
2. PostgreSQL을 업무 데이터의 기준 원장으로 확립한다.
3. 수주→생산 요청→자재 확인→출고의 핵심 흐름을 시스템에서 처리한다.
4. 부서·역할·기능 단위 권한과 감사 로그를 적용한다.
5. Excel은 이관·보조 입력·내보내기 수단으로 제한하고 의존도를 단계적으로 낮춘다.
6. 운영자가 동기화, 오류, 사용자 활동, 시스템 상태를 확인할 수 있게 한다.

### 3.3 성공 지표

초기 운영 3개월 내 다음 지표를 목표로 한다.

| 구분                           |                    목표 |
| ------------------------------ | ----------------------: |
| 핵심 원장 데이터 이관 정합률   |              99.9% 이상 |
| 주요 화면 API 응답 시간        |          p95 500ms 이하 |
| 일반 목록 화면 표시 시간       |            p95 2초 이하 |
| 월간 가용성                    |              99.5% 이상 |
| 핵심 변경 감사 로그 기록률     |                    100% |
| 수주~출고 업무의 ERP 처리 비율 |                80% 이상 |
| Excel 수기 중복 입력           | 기존 대비 50% 이상 감소 |
| 치명적 권한 오류               |                     0건 |

## 4. 제품 원칙

1. **업무 흐름 우선**: 메뉴별 기능보다 부서 간 인계가 끊기지 않는 것을 우선한다.
2. **PostgreSQL 단일 기준 원장**: 전환 완료 후 업무 상태의 최종값은 DB에 저장한다.
3. **모든 변경은 추적 가능**: 누가, 언제, 무엇을, 왜 변경했는지 기록한다.
4. **권한은 서버에서 강제**: 화면 숨김만으로 권한을 처리하지 않는다.
5. **점진 전환**: 기존 Streamlit과 Excel을 즉시 제거하지 않고 검증 후 단계적으로 종료한다.
6. **복구 가능한 자동화**: 동기화·문서 생성 실패 시 기존 정상 데이터를 보존한다.
7. **한국 업무 환경 최적화**: 한국어 UI, 원화, 국내 날짜·문서 양식을 기본으로 한다.

## 7. 핵심 업무 흐름

### 7.1 수주에서 출고까지

```text
견적 등록/문서 추출
  → 수주 확정
  → 생산 요청
  → BOM 기준 자재 소요 계산
  → 부족 자재 구매 요청
  → 생산 진행
  → 검수 완료
  → 출고 요청/승인
  → 재고 차감 및 출고 완료
```

핵심 정책:

- 수주번호, 생산요청번호, 출고번호는 서버가 발급한다.
- 상태 전이는 허용된 이전 상태에서만 가능하다.
- 확정 이후 수량·금액·납기 변경에는 사유와 이력을 남긴다.
- 출고 완료와 재고 이동은 하나의 DB 트랜잭션으로 처리한다.
- 부분 생산·부분 출고를 지원한다.

### 7.2 구매와 입고

```text
부족 자재 확인/구매 요청
  → 승인
  → 발주서 생성
  → 공급업체 발주
  → 부분/전체 입고
  → 자재 재고 증가
  → 발주 종결
```

### 7.3 AS

```text
AS 접수
  → 담당자 배정
  → 증상 확인
  → 수리/교체/반품 처리
  → 비용 및 사용 부품 기록
  → 고객 확인
  → 종결
```

## 9. 데이터 모델 초안

모든 업무 테이블은 원칙적으로 `id(UUID)`, `created_at`, `created_by`, `updated_at`, `updated_by`, `version`을 가진다. 삭제가 필요한 주요 데이터는 물리 삭제보다 상태 변경 또는 soft delete를 사용한다.

### 9.1 조직·사용자

- `departments`: 부서
- `users`: 이메일, 이름, 상태, 비밀번호 해시 또는 SSO 식별자
- `roles`, `permissions`, `user_roles`, `role_permissions`
- `audit_logs`: 행위자, 액션, 대상, 이전값, 이후값, IP, user-agent, 시각

### 9.2 기준정보

- `customers`, `suppliers`
- `products`, `materials`, `batteries`
- `warehouses`, `storage_locations`
- `bom_headers`, `bom_items`: 제품별 리비전, 유효기간, 승인 상태
- `code_groups`, `codes`

### 9.3 영업·생산·출고

- `quotes`, `quote_items`
- `sales_orders`, `sales_order_items`, `sales_order_changes`
- `production_requests`, `production_request_items`
- `production_orders`, `production_progress_logs`
- `shipment_requests`, `shipments`, `shipment_items`

### 9.4 구매·재고

- `purchase_requests`, `purchase_request_items`
- `purchase_orders`, `purchase_order_items`
- `receipts`, `receipt_items`
- `inventory_balances`: 품목·창고·로케이션별 현재 수량 캐시
- `inventory_movements`: 입고, 출고, 이동, 조정, 반품, 폐기 원장
- `stock_reservations`: 수주·생산 요청용 예약 재고

재고의 진실 원장은 `inventory_movements`이며 `inventory_balances`는 트랜잭션 안에서 함께 갱신하는 조회 최적화 값으로 정의한다. 수량은 `NUMERIC`, 금액은 `NUMERIC(18,2)`를 사용하고 float를 사용하지 않는다.

### 9.5 AS·문서·이관

- `service_cases`, `service_case_actions`, `service_parts_used`
- `documents`, `document_versions`, `document_templates`
- `attachments`: 메타데이터만 DB에 저장하고 실제 파일은 객체 스토리지에 저장
- `import_jobs`, `import_job_errors`, `source_mappings`
- `notifications`, `outbox_events`

## 10. 상태 모델

| 영역 | 상태 예시                                                                                            |
| ---- | ---------------------------------------------------------------------------------------------------- |
| 수주 | `draft → confirmed → in_production → ready_to_ship → partially_shipped → shipped → closed/cancelled` |
| 생산 | `requested → accepted → planned → in_progress → inspection → completed/cancelled`                    |
| 구매 | `draft → requested → approved → ordered → partially_received → received → closed/cancelled`          |
| 출고 | `requested → approved → picking → ready → shipped/cancelled`                                         |
| AS   | `received → assigned → diagnosing → repairing → waiting_parts → completed → closed/cancelled`        |

상태 전이 규칙은 프론트엔드가 아닌 백엔드 도메인 서비스에서 검증한다.

## 11. 시스템 아키텍처

```text
React + TypeScript
  └─ HTTPS / REST API
      FastAPI
      ├─ 인증·권한
      ├─ 업무 도메인 서비스
      ├─ 문서 생성 작업
      ├─ Excel 이관 작업
      └─ 감사/운영 로그
          ├─ PostgreSQL
          ├─ Redis (작업 큐·캐시, 필요 시)
          └─ S3 호환 객체 스토리지 (첨부·산출물)
```

## 13. 인증과 보안

- 1차는 앱 자체 이메일/비밀번호 인증을 유지하되 향후 Microsoft/Google/Cloudflare SSO를 연결할 수 있게 설계한다.

- 운영 비밀정보는 환경변수 또는 secret manager에 저장

- 개인정보·고객정보의 로그 출력 금지 및 최소 권한 적용

## 14. Excel·SQLite 전환 전략

### 14.1 전환 원칙

기존 Excel 및 SQLite 데이터를 그대로 PostgreSQL 테이블로 복사하지 않는다. 먼저 업무 개념, 식별자, 필수값, 코드값을 표준화한 뒤 staging 영역을 거쳐 적재한다.

```text
Excel / SQLite
  → 원본 보관(raw)
  → staging 테이블
  → 형식·중복·참조 검증
  → 도메인 테이블 반영
  → 건수·금액·수량 대사
  → 오류 보고서
```

### 14.2 식별자 전략

- 내부 PK는 UUID
- 기존 주문번호, 제품코드, 제조지시서 번호 등은 `legacy_key` 또는 업무번호로 보존
- 원본 파일명, 시트명, 행번호, 원본 해시를 이관 추적 정보로 저장
- 자연키가 불안정한 데이터는 매핑 테이블로 연결하고 임의 병합 금지

### 14.3 단계별 전환

1. 전체 원본 백업과 컬럼 사전 작성
2. 데이터 프로파일링: 결측, 중복, 형식, 코드 변형 분석
3. 샘플 이관과 사용자 대사
4. 읽기 전용 병행 운영
5. 도메인 단위로 PostgreSQL 쓰기 전환
6. 최종 증분 이관 및 cutover
7. Streamlit을 일정 기간 조회 전용으로 유지
8. 안정화 후 레거시 종료

### 14.4 필수 대사 항목

- 테이블/시트별 전체 건수
- 수주·구매·재고·AS의 기간별 건수
- 금액 합계와 수량 합계
- 미출고, 현재고, 미종결 AS 등 핵심 상태별 건수
- 샘플 업무번호의 상세 필드 일치 여부
- 이관 제외 및 오류 행 목록과 담당자 승인

## 15. 비기능 요구사항

### 15.1 성능·동시성

- 초기 목표 동시 사용자 30명, 향후 100명까지 수평 확장 가능

### 15.3 관측성

- 애플리케이션 로그, API 오류, 작업 큐 실패, DB 상태 모니터링
- 요청별 correlation ID
- 로그인 실패, 권한 거부, 재고 조정, 금액 변경, 내보내기 알림
- 운영 대시보드에 배포 버전과 최근 작업 실패 표시

### 15.4 접근성·호환성

- 최신 Chrome과 Edge 우선 지원
- 키보드 조작, 명확한 focus, 라벨, 충분한 색 대비
- 데스크톱 우선 반응형 UI, 현장 조회를 위한 태블릿 대응

## 18. 운영과 배포

## 부록 A. MVP 대표 인수 시나리오

### A-1. 수주 등록과 생산 요청

- 영업 담당자가 수주를 등록하고 품목, 수량, 납기를 입력한다.
- 확정 시 시스템이 업무번호를 발급한다.
- 생산 요청을 생성하면 생산팀의 내 할 일에 표시된다.
- 변경 이력에 생성자, 시각, 이전·이후 상태가 기록된다.

### A-2. 부분 출고

- 수주 수량 10개 중 6개를 먼저 출고한다.
- 재고 이동 원장에 6개 출고가 기록되고 현재고가 감소한다.
- 수주는 `partially_shipped`, 잔여 미출고는 4개로 표시된다.
- 동일 요청을 재전송해도 중복 차감되지 않는다.

### A-3. Excel 이관 오류

- 필수 제품코드가 없는 행은 운영 테이블에 반영하지 않는다.
- 작업은 성공·경고·실패 건수를 보여준다.
- 오류 보고서에서 파일, 시트, 행, 컬럼, 사유를 확인할 수 있다.
- 수정 후 재실행해도 기존 정상 데이터가 중복 생성되지 않는다.

### A-4. 권한 거부

- 일반 직원은 사용자 관리 메뉴와 API에 접근할 수 없다.
- URL 직접 입력 또는 API 호출 시에도 403을 반환한다.
- 권한 거부 이벤트는 보안 로그에 기록된다.

## 부록 B. 현행 자산 재사용 판단

| 현행 자산              | 판단               | 활용 방식                                 |
