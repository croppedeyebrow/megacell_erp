# MegaCell ERP AI 개발 정책 및 체크리스트

문서 버전: v0.1  
작성일: 2026-07-11  
적용 대상: MegaCell ERP 2.0의 기획, 설계, 구현, 리뷰, 테스트, 배포에 참여하는 AI와 개발자

## 1. 목적

이 문서는 AI가 개발 과정에서 임의로 업무 정책을 만들거나 데이터 정합성·보안·운영 원칙을 훼손하지 않도록 하는 상시 점검 기준이다.

요구사항 문서에 없는 정책을 구현해야 할 때는 이 문서의 의사결정 규칙을 따른다. 이 문서는 코드 스타일 가이드가 아니라 **제품 및 업무 안전장치**다.

## 2. 지시 우선순위

충돌이 발생하면 다음 순서로 판단한다.

1. 회사에서 명시적으로 승인한 업무·보안 정책
2. 제품 기획서와 본 정책 문서
3. 데이터 모델·API·UI 설계 문서
4. 승인된 이슈와 인수 조건
5. 기존 코드의 동작
6. AI 또는 개발자의 추정

상위 정책과 기존 구현이 충돌하면 기존 동작을 그대로 답습하지 않는다. 충돌 내용을 기록하고 승인을 요청한다.

## 3. AI 작업 절차

### 3.1 작업 전

- [ ] 요청의 대상 사용자와 업무 목적을 확인했다.
- [ ] 관련 제품 기획, 정책, 아키텍처, UI 문서를 확인했다.
- [ ] 변경되는 도메인, 테이블, API, 화면, 권한을 식별했다.
- [ ] 기존 데이터 및 레거시 동작과의 호환 영향을 확인했다.
- [ ] 요구사항에 없는 업무 정책을 추정하고 있지 않은지 확인했다.
- [ ] 완료 조건과 검증 방법을 정의했다.

### 3.2 작업 중

- [ ] 변경 범위를 요청된 기능 안으로 제한했다.
- [ ] 업무 규칙을 프론트엔드에만 구현하지 않았다.
- [ ] 중요한 변경을 트랜잭션으로 보호했다.
- [ ] 권한을 API에서 검증했다.
- [ ] 감사 로그 대상 작업을 누락하지 않았다.
- [ ] 실패·재시도·중복 요청을 고려했다.
- [ ] 기존 사용자 변경사항이나 무관한 코드를 덮어쓰지 않았다.

### 3.3 작업 후

- [ ] 정상·실패·권한 거부·경계값 테스트를 수행했다.
- [ ] DB 마이그레이션과 기존 데이터 영향을 검증했다.
- [ ] API 계약과 UI 상태가 일치하는지 확인했다.
- [ ] 문서 또는 인수 조건을 갱신했다.
- [ ] 알려진 제약과 미확정 정책을 명시했다.
- [ ] 배포 및 복구 가능성을 확인했다.

## 4. 범위와 의사결정 정책

### 4.1 AI가 독자적으로 결정해도 되는 사항

- 기존 아키텍처 규칙 안의 파일·모듈 배치
- 명확한 요구사항을 충족하기 위한 내부 구현 방식
- 사용자 동작을 바꾸지 않는 리팩터링
- 테스트, 타입, 로깅, 오류 처리 보강
- 기존 디자인 시스템 안의 컴포넌트 조합

### 4.2 반드시 확인이 필요한 사항

- 업무 상태 또는 상태 전이 추가·삭제
- 수주, 재고, 금액, 납기, 승인 정책 변경
- 재고 음수 허용, 취소, 반품, 부분 출고 정책
- 사용자 역할·권한 범위 확대
- 데이터 삭제·병합·중복 판단 기준
- Excel과 PostgreSQL 중 어느 쪽을 기준 원장으로 할지 변경
- 고객정보·문서 보존 기간 변경
- 외부 서비스, 메시지 발송, 결제, 회계 연동
- 운영 데이터의 대량 수정이나 파괴적 마이그레이션

### 4.3 미확정 정책 처리

미확정 사항은 임의로 확정하지 않고 다음 형식으로 기록한다.

```text
[POLICY-OPEN]
주제:
현재 확인된 사실:
선택지:
각 선택지의 영향:
권고안:
결정권자:
결정 필요 시점:
```

개발을 막지 않는 부분은 feature flag 또는 보수적인 기본값으로 격리한다. 금액·재고·권한·삭제에 영향을 주는 미확정 사항은 구현을 확정하지 않는다.

## 5. 도메인 정책

### 5.1 공통

- 업무번호는 서버에서 생성하며 사용자가 임의로 중복 생성할 수 없다.
- 상태 전이는 백엔드 도메인 서비스가 검증한다.
- 확정된 업무 데이터의 중요 필드 변경에는 사유를 받는다.
- 주요 업무 데이터는 물리 삭제보다 취소 또는 비활성 상태를 사용한다.
- 모든 시각은 DB에 timezone-aware 형식으로 저장하고 UI에서 Asia/Seoul로 표시한다.
- 금액과 수량 계산에 부동소수점 타입을 사용하지 않는다.
- 서버가 계산해야 하는 합계와 상태를 클라이언트 입력값으로 신뢰하지 않는다.

### 5.2 수주

- 수주 확정 후 품목, 수량, 금액, 납기 변경은 변경 전·후 값과 사유를 기록한다.
- 부분 출고를 지원하며 출고량 합계는 수주량을 초과할 수 없다.
- 취소된 수주는 새 생산·출고 요청을 만들 수 없다.
- 미출고량은 수주량과 유효 출고량을 기준으로 서버가 계산한다.
- 수주번호 또는 legacy key 중복 시 자동 병합하지 않는다.

### 5.3 생산

- 생산 요청은 유효한 수주 또는 승인된 내부 요청과 연결한다.
- 생산 완료 수량은 요청 수량을 초과할 수 없다. 초과 생산 정책은 별도 승인 사항이다.
- 생산 진행 변경은 담당자, 변경 시각, 수량, 메모를 기록한다.
- BOM은 적용 리비전을 명시해야 하며 과거 생산 기록의 리비전을 소급 변경하지 않는다.

### 5.4 재고와 출고

- 재고의 진실 원장은 `inventory_movements`다.
- `inventory_balances`는 이동 원장과 동일 트랜잭션에서 갱신한다.
- 출고 완료와 재고 차감은 하나의 원자적 트랜잭션이다.
- 재고 조정에는 유형, 수량, 사유, 작업자, 승인자를 기록한다.
- 음수 재고는 기본적으로 금지한다. 허용 여부는 명시적 정책 승인 대상이다.
- 중복 출고 방지를 위해 idempotency key 또는 고유 업무키를 사용한다.
- 과거 이동 내역을 직접 수정하지 않고 반대 이동 또는 조정 내역을 추가한다.

### 5.5 구매·입고

- 발주는 승인된 구매 요청 또는 명시적 예외 사유와 연결한다.
- 부분 입고를 허용하고 누적 입고량은 발주량을 초과하지 않는다.
- 초과 입고, 단가 변경, 발주 취소는 승인 정책 대상으로 처리한다.
- 입고 완료와 자재 재고 증가는 하나의 트랜잭션으로 처리한다.

### 5.6 AS

- AS 접수에는 고객, 제품 또는 식별 가능한 대체 정보, 증상, 접수일이 필요하다.
- 사용 부품과 배터리는 재고 이동과 연결한다.
- 종결 시 처리 결과, 처리일, 담당자를 필수로 기록한다.
- AS 이력은 원칙적으로 삭제하지 않는다.

### 5.7 문서

- PDF 자동 추출 결과는 사용자 확인 전 확정 데이터로 저장하지 않는다.
- 원본 파일, parser 버전, 템플릿 버전, 생성자, 생성 시각을 기록한다.
- 문서 추출 실패는 기존 업무 데이터를 변경하지 않는다.
- 생성 파일명만으로 고객이나 업무를 식별하지 않고 DB 관계를 사용한다.

## 6. 데이터 정책

### 6.1 기준 원장

- 신규 시스템 전환 완료 후 PostgreSQL이 업무 데이터의 최종 기준이다.
- Excel은 초기 이관, 제한된 보조 입력, 내보내기에 사용한다.
- 병행 운영 기간에는 도메인마다 쓰기 주체를 하나로 지정한다.
- 양쪽 시스템에서 동시에 같은 데이터를 수정하는 양방향 동기화는 원칙적으로 금지한다.

### 6.2 데이터 무결성

- 참조 무결성은 가능한 한 DB foreign key로 강제한다.
- 필수값, unique, check constraint를 애플리케이션 검증과 함께 적용한다.
- DB 마이그레이션은 Alembic으로 관리한다.
- 운영 DB 스키마를 수동으로 변경하지 않는다.
- 모든 마이그레이션은 기존 데이터가 있는 상태로 테스트한다.

### 6.3 이관

- 원본은 raw 상태로 보존하고 staging을 거쳐 검증한다.
- 원본 파일, 시트, 행번호, 해시, legacy key를 추적 가능하게 저장한다.
- 오류 행은 묵시적으로 버리지 않고 오류 보고서에 포함한다.
- 이관 작업은 재실행 가능하고 중복 결과를 만들지 않아야 한다.
- 건수, 수량, 금액, 핵심 상태별 대사를 통과해야 운영 반영한다.

### 6.4 삭제와 보존

- 사용자 화면에서 삭제가 필요해도 업무 데이터는 기본적으로 soft delete 또는 취소 처리한다.
- 감사 로그는 일반 관리자가 수정·삭제할 수 없다.
- 개인정보, 문서, 로그 보존 기간은 회사 정책 확정 전 임의로 정하지 않는다.

## 7. API 정책

- 모든 업무 API는 `/api/v1` 아래에 둔다.
- 요청·응답 스키마를 명시하고 OpenAPI와 실제 동작을 일치시킨다.
- 목록 API는 페이지네이션, 필터, 정렬을 제공한다.
- 무제한 전체 목록 응답을 만들지 않는다.
- 오류는 안정적인 error code, 사용자 메시지, field errors, correlation ID를 제공한다.
- 수정 충돌 방지를 위해 version 또는 ETag 기반 낙관적 잠금을 사용한다.
- 네트워크 재시도 가능한 생성·완료 API는 idempotency를 지원한다.
- 내부 예외, SQL, 경로, 비밀정보를 응답에 노출하지 않는다.
- API breaking change는 버전 또는 명시적 마이그레이션 절차 없이 적용하지 않는다.

## 8. 인증·권한·보안 정책

- 인증과 권한 검사는 모든 보호 API에서 서버 측으로 수행한다.
- UI 메뉴 숨김은 편의 기능일 뿐 보안 통제가 아니다.
- 권한은 deny-by-default로 설계한다.
- 비밀번호는 Argon2id를 사용하고 평문·복호화 가능 형태로 저장하지 않는다.
- 인증 쿠키 사용 시 HttpOnly, Secure, SameSite 및 CSRF 방어를 적용한다.
- 로그인 시도 제한과 계정 비활성화를 지원한다.
- 비밀정보를 코드, Git, 로그, 프론트엔드 번들에 포함하지 않는다.
- 고객정보, 이메일, 문서 내용 등 민감정보를 일반 로그에 기록하지 않는다.
- 데이터 내보내기와 문서 다운로드에도 권한과 감사 로그를 적용한다.
- 파일 업로드는 확장자, MIME, 크기, 파일명, 저장 경로를 검증한다.

## 9. 감사 로그 정책

다음 작업은 감사 로그 필수 대상이다.

- 로그인 성공·실패, 로그아웃, 비밀번호 변경
- 사용자·역할·권한 생성 및 변경
- 수주 금액·수량·납기·상태 변경
- 생산·구매·출고·AS 상태 전이
- 재고 입출고·조정·반품·폐기
- 문서 생성·다운로드·삭제 상태 변경
- Excel 이관·동기화 실행과 결과
- 대량 내보내기
- 관리자 설정 변경

감사 로그에는 행위자, 시각, action, 대상 유형과 ID, 변경 전·후 값, 사유, 요청 ID를 기록한다. 비밀번호, 토큰, 파일 본문 등 비밀값은 기록하지 않는다.

## 10. UI 정책

- 저장·승인·출고·취소 등 결과가 다른 버튼을 명확히 구분한다.
- 위험 작업은 대상과 결과를 설명하고 확인을 받는다.
- 서버 응답 전에 성공한 것처럼 최종 상태를 표시하지 않는다.
- 로딩, 빈 결과, 권한 없음, 오류, 부분 성공 상태를 각각 설계한다.
- 색상만으로 상태나 오류를 전달하지 않는다.
- 표의 기본 정렬·필터 상태를 URL에 반영해 공유와 복귀가 가능하게 한다.
- 사용자 입력이 남아 있는 화면에서 이탈 경고를 제공한다.
- 권한 없는 동작은 숨기거나 비활성화하되 API에서도 거부한다.

## 11. 테스트 정책

기능 추가 시 최소 다음 테스트를 고려한다.

- 정상 흐름
- 필수값 누락과 잘못된 형식
- 권한 없음
- 존재하지 않거나 이미 변경된 대상
- 중복 요청과 재시도
- 동시 수정
- 트랜잭션 중간 실패
- 0, 음수, 매우 큰 수량·금액
- 부분 출고·부분 입고
- 타임존과 날짜 경계
- 기존 데이터가 있는 DB 마이그레이션

재고, 금액, 권한, 상태 전이 코드는 단위 테스트와 API 통합 테스트를 모두 작성한다.

## 12. 배포 정책

- 개발, 스테이징, 운영 환경을 분리한다.
- 운영 배포 전 lint, type check, test, build, migration check를 통과한다.
- DB 변경이 포함되면 백업 및 roll-forward/복구 방법을 준비한다.
- 배포 후 로그인, 주요 목록, DB 연결, 핵심 쓰기 흐름 smoke test를 수행한다.
- 기능 플래그가 필요한 기능은 기본 비활성 상태로 안전하게 배포한다.
- 운영 장애 시 데이터 정합성을 우선하고 부분 처리 결과를 확인한다.

## 13. 코드 리뷰 체크리스트

### 업무·데이터

- [ ] 상태 전이가 도메인 정책과 일치하는가?
- [ ] 재고·금액 계산의 기준 데이터가 명확한가?
- [ ] 중요한 쓰기가 하나의 트랜잭션인가?
- [ ] 동시 요청과 중복 실행에 안전한가?
- [ ] 변경 이력과 감사 로그가 필요한가?

### 보안

- [ ] API 권한 검사가 있는가?
- [ ] 입력과 파일을 서버에서 검증하는가?
- [ ] 민감정보가 응답·로그에 노출되지 않는가?
- [ ] 관리자 기능이 deny-by-default인가?

### 품질

- [ ] 도메인 로직이 router 또는 UI에 섞이지 않았는가?
- [ ] 오류가 사용자에게 실행 가능한 메시지로 전달되는가?
- [ ] 테스트가 실패·경계·권한 시나리오를 포함하는가?
- [ ] 문서와 API 스키마가 갱신되었는가?

## 14. Definition of Done

다음 조건을 모두 충족해야 완료로 처리한다.

- [ ] 인수 조건을 충족한다.
- [ ] 코드와 데이터 모델이 아키텍처 규칙을 따른다.
- [ ] 권한·감사 로그 정책을 충족한다.
- [ ] 자동 테스트와 필요한 수동 검증을 통과한다.
- [ ] 마이그레이션 및 기존 데이터 호환성을 확인했다.
- [ ] 로딩·빈 상태·오류·권한 없음 UI가 존재한다.
- [ ] 운영 로그와 장애 확인 경로가 있다.
- [ ] 관련 문서를 갱신했다.
- [ ] 미확정 정책과 알려진 제한을 기록했다.
