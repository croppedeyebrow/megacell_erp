# MegaCell ERP 개발 기준서

문서 버전: 2026-07-10  
대상 프로젝트: `megacell_erp`  
저장소: `https://github.com/croppedeyebrow/megacell_erp.git`

## 1. 문서 목적

이 문서는 메가셀 ERP 개발 과정에서 논의하고 확정한 방향을 실무 개발 기준으로 정리한다. 개발자는 이 문서를 기준으로 메뉴 구조, 데이터 처리 원칙, 권한 정책, 배포 방식, 단계별 개발 우선순위를 맞춘다.

현재 프로젝트는 기존 엑셀 원장을 완전히 폐기하는 시스템이 아니라, 엑셀 원장을 원본 데이터로 유지하면서 ERP 화면에서는 SQLite DB를 통해 안정적으로 조회, 수정, 권한 관리, 문서 생성을 수행하는 방향으로 개발한다.

## 2. 제품 목표

MegaCell ERP는 부서별 업무 포털이다.

목표는 다음과 같다.

- 기존 엑셀 원장 데이터를 웹 화면에서 조회한다.
- 부서별 사용자가 본인 업무에 맞는 화면으로 바로 이동할 수 있게 한다.
- ERP에서 수정되는 업무 상태는 엑셀 원본이 아니라 운영 DB에 기록한다.
- 관리자와 일반 사용자의 권한을 분리한다.
- 외부 직원도 도메인을 통해 접속하되, 앱 자체 로그인으로 접근을 통제한다.
- 문서 출력, 주문서 변환, 향후 견적/발주/입찰 문서 생성을 자동화한다.

## 3. 확정된 큰 방향

### 3.1 실행 및 배포

- 개발은 로컬 프로젝트 폴더 `C:\Users\megaPC\Desktop\megacell_erp` 기준으로 진행한다.
- GitHub 저장소로 버전 관리한다.
- 운영 서버는 Streamlit을 `0.0.0.0:8501`로 실행한다.
- 외부 접속은 `https://erp.megacell-erp.com` 도메인을 사용한다.
- Cloudflare Tunnel은 로컬 서버를 외부 도메인으로 연결하는 통로로 사용한다.
- 사용자 인증은 Cloudflare Access에만 의존하지 않고, ERP 내부 로그인/로그아웃/회원관리 기능을 기준으로 한다.

### 3.2 기술 스택

- UI: Streamlit
- 언어: Python
- DB: SQLite
- 원본 데이터: 기존 Excel 파일
- 버전관리: Git, GitHub
- 외부 접속: Cloudflare Tunnel, 커스텀 도메인

### 3.3 개발 방식

- 기능은 한 파일에 몰아넣지 않고 역할별 폴더로 분리한다.
- 화면은 `views/`, 업무 로직은 `services/`, DB 접근은 `repositories/`, 공통 기반 코드는 `core/`에 둔다.
- 데이터 원본 파일은 `data/`, SQLite DB와 산출물은 `instance/`, 배포/운영 스크립트는 `infra/`에 둔다.
- 각 폴더에는 해당 폴더의 역할을 설명하는 `README.md`를 유지한다.

## 4. 부서 및 메뉴 구조

ERP의 첫 화면은 랜딩페이지이며, 부서별 이동 카드를 제공한다.

### 4.1 경영지원팀

경영지원팀은 메가셀 기준으로 구매, 자재, 재고 출납, 전기공사 입찰, 정산, 문서 업무를 함께 담당한다.

현재 확정 메뉴:

- 재고 현황
- 입출고 기록
- 구매/발주 관리
- 전기공사 입찰/관리
- 정산/청구 관리
- 매입/매출 집계
- 거래처/계약 자료
- 문서 발행
- 사내 자료/양식

### 4.2 기술영업팀

기술영업팀은 수주 전후의 고객 대응, 견적, 미출고, 출고 요청, AS 조회 업무를 담당한다.

현재 확정 메뉴:

- 수주/견적 현황
- 미출고 현황
- 견적서/주문서 변환
- 고객/거래처 조회
- 출고 요청
- AS 이력

### 4.3 생산팀

생산팀은 생산 접수, 생산 진행, BOM 소요량, 부족 자재 확인, 검수/출고 준비를 담당한다.

현재 확정 메뉴:

- 생산 요청 현황
- 생산 예정/진행
- BOM 소요량 확인
- 부족 자재 확인
- 검수/출고 준비

### 4.4 연구소

연구소는 제품/BOM 기준정보, 자재/배터리 마스터, 제품 사양 변경 이력, 기술 검토 자료를 담당한다.

현재 확정 메뉴:

- 제품/BOM 관리
- 자재/배터리 마스터
- 제품 사양 변경 이력
- 기술 검토 자료

### 4.5 전체 현황

전체 현황은 관리자 중심 화면이다.

현재 확정 메뉴:

- 업무 대시보드
- 경영 요약
- 수요량 분석
- 사용자 관리
- 시스템 관리

## 5. 권한 및 사용자 관리

### 5.1 인증 방식

ERP는 내부 로그인 기능을 사용한다.

- 이메일/비밀번호 기반 로그인
- 로그아웃 기능 제공
- 초기 비밀번호 또는 재설정 비밀번호 로그인 시 비밀번호 변경 강제
- 비밀번호는 평문 저장 금지
- PBKDF2 기반 해시 저장

### 5.2 사용자 테이블

사용자 정보는 SQLite `users` 테이블에 저장한다.

주요 관리 항목:

- 이메일
- 이름
- 부서
- 권한 역할
- 문서 생성 권한
- 활성/비활성 상태
- 비밀번호 변경 필요 여부

### 5.3 권한 원칙

- 관리자는 모든 메뉴 접근 가능
- 일반 사용자는 본인 부서 메뉴 중심으로 접근
- 문서 생성 기능은 별도 권한이 있는 사용자만 사용
- 사용자 관리는 관리자만 접근
- 시스템 관리와 DB 동기화는 관리자만 접근

## 6. 데이터 처리 원칙

### 6.1 엑셀 원본과 ERP 운영 DB 분리

ERP DB는 두 종류의 테이블을 함께 사용한다.

원본 적재 테이블:

- `수주`
- `재고`
- `재고출납`
- `BOM`
- `자재재고`
- `발주현황`
- `배터리*`
- `AS관리`

운영 테이블:

- `users`
- `erp_orders`
- `erp_order_change_logs`
- `erp_inventory_movements`
- `erp_documents`
- `erp_sync_state`

원본 적재 테이블은 엑셀에서 다시 만들 수 있는 조회용 데이터다. ERP 화면에서 사용자가 직접 수정하는 값은 원본 적재 테이블에 저장하지 않는다.

운영 테이블은 ERP에서 실제로 변경되는 상태를 저장한다.

### 6.2 수주 데이터 운영 원칙

`수주` 테이블은 엑셀 원본이다.

`erp_orders`는 ERP 운영 수주 원장이다.

엑셀 원본에서 최초 동기화할 때는 `erp_orders`에 값을 복사한다. 이후 ERP에서 출고 처리, 금액 수정, 출고일 수정 등을 하면 `erp_orders.erp_modified`를 켠다. 이 값이 켜진 행은 엑셀 원본을 다시 적재해도 ERP에서 수정한 값을 덮어쓰지 않는다.

변경 발생 시 `erp_order_change_logs`에 이전값, 변경값, 변경자, 변경시각, 변경 사유를 기록한다.

### 6.3 재고 데이터 운영 원칙

현재 재고 조회는 기존 엑셀 적재 데이터를 기반으로 한다.

향후 재고 증감이 ERP에서 직접 발생하면 `erp_inventory_movements`에 입고, 출고, 조정, 반품, 폐기 등 재고 이동 이력을 기록한다.

현재고는 단순 테이블 값만 보는 것이 아니라, 기준 재고와 이동 이력을 조합해 계산하는 방향으로 발전시킨다.

## 7. 현재 구현된 주요 기능

### 7.1 앱 구조

- `app.py`: 앱 진입점, 로그인 분기, 부서/메뉴 라우팅

### 7.2 로그인

- ERP 내부 로그인 화면
- 비밀번호 변경 화면
- 로그아웃 버튼
- 관리자용 사용자 관리 화면

### 7.3 DB 정리

- 원본 엑셀 적재 테이블과 ERP 운영 테이블 분리
- `erp_orders` 동기화 스크립트 추가
- `init_db.py` 실행 시 운영 테이블 보존 후 원본 재적재 가능

### 7.4 배포

- 로컬 Streamlit 서버 실행
- Cloudflare Tunnel을 통한 외부 접속
- GitHub push 후 운영 PC에서 pull 및 서버 재시작
- 백그라운드 실행 스크립트 사용 가능

## 13. 개발 판단 기준

개발 중 판단이 필요한 경우 다음 원칙을 따른다.

- 원본 엑셀은 조회/동기화 기준으로 둔다.
- ERP에서 수정되는 업무 상태는 운영 테이블에 저장한다.
- 부서 메뉴는 실제 담당 부서 기준으로 배치한다.
- 일반 직원은 조회 중심, 권한 있는 사용자는 문서 생성 가능, 관리자는 모든 권한을 가진다.
- 기능을 만들기 전에 DB 저장 위치와 변경 이력 필요 여부를 먼저 결정한다.
- 단기적으로는 Streamlit 유지, 장기적으로 React/FastAPI 등 분리형 구조로 이전 가능성을 열어둔다.
