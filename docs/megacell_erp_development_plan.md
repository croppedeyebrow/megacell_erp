# Megacell ERP 개발 기획서

작성일: 2026-06-26  
목표: 엑셀 기반 업무 원장을 안전하게 조회하고, 일부 문서 생성 업무를 자동화하는 사내 ERP 구축

## 1. 개발 방향 요약

현재 ERP는 처음부터 완전한 입력형 ERP로 만들기보다, 먼저 `조회 중심 ERP`로 안정화하는 것이 적합하다.

초기 목표는 다음 네 가지다.

1. 사내/외부 직원이 영업, 구매자재, 재고, 배터리, AS 내역을 안정적으로 조회한다.
2. 기존 엑셀 원본은 유지하되, ERP는 엑셀 데이터를 DB에 적재해서 빠르고 일관되게 보여준다.
3. PDF 견적서/발주서를 주문서, 생산지시서 엑셀로 생성하는 기능은 제한된 사용자에게만 허용한다.
4. 관리자는 데이터 갱신, 사용자 권한, 원본 파일 상태를 관리할 수 있다.

## 2. 실행 및 배포 방식

### 2.1 추천 방식

초기에는 `회사 내부 PC 또는 미니 서버 + Cloudflare Tunnel + Access` 방식으로 운영한다.

```text
직원 브라우저
  -> https://erp.megacell.co.kr
  -> Cloudflare Access 로그인
  -> Cloudflare Tunnel
  -> 사내 ERP 서버의 Streamlit 앱
  -> 로컬 DB
  -> 엑셀 원본 파일
```

이 방식은 공유기 포트 개방 없이 외부 접속을 제공할 수 있고, 현재 Streamlit 기반 개발 구조를 크게 바꾸지 않아도 된다.

### 2.2 단계별 운영 방식

| 단계 | 방식 | 목적 |
|---|---|---|
| 1단계 | 로컬 PC 실행 | 개발 및 내부 테스트 |
| 2단계 | 사내 고정 PC/미니 서버 실행 | 직원 조회 테스트 |
| 3단계 | Cloudflare Tunnel + Access | 외부 직원 조회 |
| 4단계 | PostgreSQL/클라우드 검토 | 입력/동시사용 증가 시 확장 |

초기에는 SQLite를 사용하고, ERP에서 직접 입력하는 기능이 늘어나면 PostgreSQL로 전환한다.

## 3. 데이터 적재 안정화

### 3.1 기본 원칙

엑셀은 원본으로 유지하고, ERP는 DB를 조회한다.

```text
엑셀 원본 수정
  -> 데이터 갱신 버튼 또는 주기적 동기화
  -> DB 테이블 갱신
  -> ERP 화면 조회
```

처음부터 완전 실시간 파일 감시를 적용하면 엑셀 저장 중 파일 잠김, 시트 구조 변경, 임시파일 문제로 오류가 날 수 있다. 따라서 다음 순서로 진행한다.

1. 수동 갱신 버튼
2. 주기적 자동 갱신
3. 파일 변경 감지 자동 갱신

### 3.2 원본 파일과 DB 매핑

| 원본 | ERP 영역 | DB 역할 |
|---|---|---|
| 주문현황 xlsm | 영업관리 | 수주, 납품, 미출고 |
| 자재리스트 관리 양식 xlsm | 구매-자재관리 | 자재, BOM |
| 간편재고관리프로그램 xlsm | 재고관리 | 제품 재고, 재고 출납 |
| 배터리관리 xlsb | 재고관리 | 배터리 현재고, 입출고, 사용이력 |
| AS관리대장 | AS관리 | AS 접수, 처리, 거래처별 이력 |

### 3.3 동기화 로그

관리자 화면에서 다음 정보를 확인할 수 있어야 한다.

- 원본 파일 경로
- 마지막 수정 시간
- 마지막 DB 반영 시간
- 반영된 행 수
- 실패 사유
- 이전 성공 시점

## 4. 업무 범위 및 메뉴 구조

### 4.1 최종 메뉴 구조

```text
대시보드

영업관리
  - 영업내역
  - 수주/미출고 현황
  - 견적/발주서 변환

구매-자재관리
  - 자재 현황
  - BOM 조회
  - BOM 대수 반영
  - 발주 필요 품목

재고관리
  - 제품 재고
  - 재고 출납기록
  - 배터리 재고
  - 배터리 입출고
  - 미출고 달력

AS관리
  - AS 이력
  - 거래처별 AS 조회

관리자
  - 데이터 갱신
  - 원본 파일 상태
  - 동기화 로그
  - 사용자 권한
```

### 4.2 문서 생성 위치

기존 `문서출력` 메뉴는 독립 메뉴보다 `영업관리 > 견적/발주서 변환`으로 이동하는 것이 적합하다.

문서 생성 기능은 영업 흐름 안에 있으며, 모든 직원에게 열기보다 지정된 사용자에게만 허용한다.

## 5. 권한 설계

### 5.1 권한 그룹

| 권한 | 대상 | 가능 기능 |
|---|---|---|
| viewer | 일반 직원 | 전체 조회 |
| document_user | 문서 담당자 | 조회, PDF 업로드, 주문서/생산지시서 생성 |
| admin | 관리자 | 전체 조회, 문서 생성, 데이터 갱신, 권한 관리 |

### 5.2 초기 인증 방식

Cloudflare Access에서 로그인한 이메일을 기준으로 권한을 매핑한다.

```text
admin@megacell.co.kr -> admin
sales@megacell.co.kr -> document_user
company-domain users -> viewer
```

Streamlit 앱 내부에서는 현재 사용자 이메일을 읽어서 메뉴, 버튼, 다운로드 기능을 제한한다.

### 5.3 권한 적용 우선순위

1. 관리자 메뉴 숨김
2. 데이터 갱신 버튼 제한
3. PDF 업로드/문서 생성 제한
4. DB 수정 기능 제한
5. 사용자별 접근 로그 저장

목표: 현재 개발 파일을 보존하고, 이후 변경을 추적할 수 있게 만든다.

작업:

- 현재 `megacell_erp` 폴더 백업
- Git 저장소 초기화
- `.gitignore` 작성
- README 작성
- 실행 방법 문서화
- 현재 기능 목록 정리

완료 기준:

- 새 기능 개발 전에 언제든 현재 상태로 돌아갈 수 있다.

### 1단계: 실행 안정화

목표: ERP 서버가 안정적으로 켜지고 꺼지며, 직원 테스트가 가능하다.

작업:

- `start_erp.bat`, `stop_erp.bat`, `erp_status.bat` 정리
- 실행 로그 파일 저장
- 서버 중복 실행 방지
- Python/패키지 버전 고정
- 로컬 접속 주소와 사내 접속 주소 안내

완료 기준:

- 더블클릭으로 서버 실행
- 상태 확인 가능
- 문제 발생 시 로그 확인 가능

### 2단계: 메뉴 및 조회 화면 정리

목표: ERP의 업무 구조를 고정한다.

작업:

- 메뉴를 영업관리, 구매-자재관리, 재고관리, AS관리, 관리자로 재배치
- 문서출력 기능을 영업관리 안으로 이동
- 대시보드는 핵심 현황만 표시
- 직원 조회용 화면을 우선 정리

완료 기준:

- 사용자가 업무 메뉴를 보고 위치를 바로 이해할 수 있다.

### 3단계: 데이터 적재 구조 정리

목표: 엑셀 원본과 DB 테이블의 관계를 명확히 한다.

작업:

- 원본 파일별 적재 함수 분리
- DB 테이블 구조 정리
- 동기화 로그 테이블 추가
- 관리자 화면에 수동 갱신 버튼 추가
- 적재 실패 메시지 개선

완료 기준:

- 어떤 엑셀이 어떤 화면의 원본인지 추적 가능하다.

### 4단계: 권한 기능 추가

목표: 전체 직원은 조회만, 문서 담당자는 생성 가능, 관리자는 전체 권한을 갖는다.

작업:

- 권한 그룹 정의
- 사용자 이메일 기반 권한 매핑 파일 추가
- 메뉴/버튼 단위 권한 적용
- 문서 생성 다운로드 권한 제한
- 관리자 기능 제한

완료 기준:

- 일반 직원 계정으로 접속하면 조회 외 기능이 보이지 않는다.

### 5단계: 문서 변환 안정화

목표: PDF 견적서/발주서에서 주문서 xlsm을 안정적으로 생성한다.

작업:

- 견적서/발주서 자동 판별 개선

- 추출 결과 수정 후 생성
- 생성 이력 저장
- 템플릿 파일 상태 점검

완료 기준:

- 담당자가 PDF를 넣고 주문서 xlsm을 생성할 수 있다.

### 6단계: 외부 접속 PoC

목표: Cloudflare Tunnel + Access로 외부 직원 조회를 테스트한다.

작업:

작업:

완료 기준:

- 엑셀 수정 후 일정 시간 내 ERP에 반영된다.

### 8단계: 입력형 ERP 확장 검토

목표: 일부 업무를 엑셀에서 ERP 직접 입력으로 전환한다.

우선순위:

1. 비고/세부내역 수정
2. 미출고/출고완료 상태 수정
3. AS 접수 입력
4. 문서 생성 이력 관리
5. 재고 입출고 입력

완료 기준:

- 엑셀 의존도가 줄고, ERP가 원장 역할을 일부 수행한다.

## 7. Git 관리 기획

### 7.1 Git을 도입해야 하는 이유

현재처럼 로컬 파일만 수정하면 다음 문제가 생긴다.

- 어떤 파일을 언제 왜 고쳤는지 추적하기 어렵다.
- 기능이 깨졌을 때 정상 상태로 되돌리기 어렵다.
- 테스트용 수정과 실제 운영 수정이 섞인다.
- 다른 PC나 서버로 옮길 때 누락 파일이 생기기 쉽다.

Git을 도입하면 개발 이력, 백업, 배포 기준을 명확히 관리할 수 있다.

### 7.2 저장소 위치

추천은 `C:\Users\megaPC\Desktop\megacell_erp` 폴더 자체를 Git 저장소로 만드는 것이다.

```text
C:\Users\megaPC\Desktop\megacell_erp
  .git
  app.py
  init_db.py
  requirements.txt
  start_erp.bat
  stop_erp.bat
  docs
  templates
```

### 7.3 Git에 포함할 파일

포함:

- Python 소스 코드
- 실행 bat 파일
- requirements.txt
- README
- 문서 템플릿 샘플
- 설정 샘플 파일
- 개발 기획서

제외:

- 실제 DB 파일
- 실제 엑셀 원본
- 고객 PDF
- 생성된 주문서/생산지시서
- 로그 파일
- 비밀번호/토큰/Cloudflare 인증정보
- 백업 zip

### 7.5 브랜치 전략

초기에는 단순하게 운영한다.

```text

  - 실제 운영 가능한 안정 버전

dev
  - 개발 통합 버전

feature/*
  - 기능별 작업 브랜치
```

작업 예시:

```text
feature/auth-permissions

feature/document-converter
```

### 7.6 커밋 규칙

커밋은 작은 단위로 남긴다.

예시:

```text
feat: 사용자 권한 그룹 추가
fix: 재고 데이터 캐시 갱신 오류 수정
docs: ERP 개발 기획서 추가
chore: 실행 스크립트 정리
```

### 7.7 백업 저장소

가능한 선택지는 세 가지다.

| 방식 | 장점 | 단점 |
|---|---|---|
| GitHub Private | 관리 쉬움, 백업 안정적 | 외부 서비스 사용 |
| GitLab Private | 사내 개발 관리에 적합 | 초기 설정 필요 |
| NAS bare repo | 내부망 보관 | 외부 백업 안정성은 낮음 |

추천은 `GitHub Private 또는 GitLab Private + NAS 주기 백업`이다.

단, 고객 문서, 실제 DB, 엑셀 원본은 Git에 올리지 않는다.

## 9. 개발 판단 기준

앞으로 기능을 추가할 때는 아래 기준을 따른다.

1. 조회 기능이 먼저다.
2. 원본 엑셀을 훼손하지 않는다.
3. DB 갱신 실패 시 기존 정상 데이터를 유지한다.
4. 일반 직원에게는 수정/생성 기능을 노출하지 않는다.
5. 관리자 기능은 로그와 복구 수단을 함께 만든다.
6. Git 커밋 없이 큰 기능을 수정하지 않는다.
7. 외부 접속은 Cloudflare Access 뒤에서만 허용한다.

## 10. 최종 목표

1차 최종 목표는 다음 상태다.

```text
직원은 외부에서 안전하게 ERP를 조회한다.
문서 담당자는 PDF로 주문서/생산지시서를 생성한다.
관리자는 엑셀 원본을 DB에 반영하고 상태를 확인한다.
개발자는 Git으로 변경 이력을 추적하고 안정 버전을 관리한다.
```

이 상태가 안정화되면, 이후 ERP 직접 입력과 PostgreSQL 전환을 검토한다.
