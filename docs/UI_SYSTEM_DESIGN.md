# MegaCell ERP 2.0 UI 시스템 설계서

문서 버전: v0.1  
작성일: 2026-07-11  
대상: React + TypeScript Web Application

## 1. 목적

이 문서는 MegaCell ERP의 화면 구조, 사용자 경험 원칙, 디자인 토큰, 공통 컴포넌트, 데이터 표시, 상태와 권한 처리 기준을 정의한다.

목표는 기능별 화면이 서로 다른 방식으로 동작하는 것을 방지하고, 사용자가 어느 부서 메뉴에서도 동일한 조작 원리를 학습하도록 만드는 것이다.

## 2. UX 목표

- 업무 담당자가 현재 상태와 다음 행동을 빠르게 파악한다.
- 검색, 필터, 표, 상세, 수정 흐름을 모든 업무에서 일관되게 제공한다.
- 중요한 금액·수량·납기·재고 변경에서 실수를 예방한다.
- 권한과 상태에 따라 가능한 행동을 명확히 보여준다.
- 실패했을 때 원인과 해결 방법을 사용자 언어로 안내한다.
- 데스크톱 중심이지만 태블릿에서도 조회와 핵심 승인이 가능하다.

## 3. 사용자 경험 원칙

1. **업무 중심 언어**: 기술 용어보다 회사에서 사용하는 업무 용어를 사용한다.
2. **상태와 다음 행동**: 상세 상단에 현재 상태, 담당자, 다음 가능한 행동을 노출한다.
3. **실수 예방**: 위험한 작업은 결과를 설명하고 사유 또는 확인을 받는다.
4. **점진적 정보 공개**: 목록에는 판단에 필요한 정보만, 상세에는 전체 이력과 연결 정보를 제공한다.
5. **일관된 피드백**: 저장·실패·경고·진행 중 상태를 동일한 패턴으로 표시한다.
6. **복구 가능성**: 입력 유실 방지, 재시도, 임시 저장 또는 취소 경로를 제공한다.
7. **접근성**: 색상만으로 의미를 전달하지 않고 키보드와 보조기술 사용을 고려한다.

## 4. 전체 정보 구조

```text
홈
영업
생산
구매·자재
재고·출고
AS
기준정보
분석
관리
```

사용자에게 권한이 없는 최상위 메뉴는 숨긴다. 단, URL 직접 접근 시 권한 없음 화면을 표시하며 API에서도 거부한다.

### 5.2 태블릿·모바일

- 사이드바는 drawer로 전환한다.
- 표는 핵심 열 우선 표시, 가로 스크롤 또는 카드 보기 제공
- 주요 액션은 화면 하단 고정 대신 header 또는 overflow 메뉴 사용
- 복잡한 대량 편집은 모바일에서 제한하고 안내할 수 있다.

## 6. 레이아웃 규칙

- 기본 콘텐츠 최대 폭: 데이터 표 화면은 넓게, 폼·설정 화면은 제한된 폭 사용
- 페이지 여백: desktop 24px, tablet 20px, mobile 16px
- 8px spacing grid 사용
- 제목과 주요 액션은 동일한 page header 영역에 배치
- 한 화면의 primary action은 원칙적으로 하나
- 필터 영역과 결과 영역을 시각적으로 분리
- 긴 상세 화면은 section과 sticky local navigation 사용 검토

## 8. 공통 컴포넌트

### 8.1 Action

- `Button`: primary, secondary, ghost, danger
- `IconButton`: tooltip과 accessible label 필수
- `SplitButton`: 주요 행동과 관련 보조 행동
- `DropdownMenu`: 빈도가 낮은 행동

버튼 상태: default, hover, focus, active, disabled, loading.

삭제·취소·재고 조정 등 위험 행동에 primary 색상을 사용하지 않는다.

### 8.2 Input

- TextField, TextArea, NumberField
- Select, MultiSelect, Combobox
- DatePicker, DateRangePicker
- Checkbox, Radio, Switch
- FileUpload
- SearchField

모든 입력은 label을 가진다. placeholder를 label 대신 사용하지 않는다. 필수 여부와 오류 메시지는 입력 가까이에 표시한다.

### 8.3 Display

- Badge/StatusBadge
- Tooltip
- DescriptionList
- StatCard
- EmptyState
- Skeleton
- Alert
- Toast
- Progress
- Timeline/ActivityLog

### 8.4 Overlay

- Modal: 명확한 확인이나 짧은 폼
- Drawer: 목록 문맥을 유지한 빠른 상세·편집
- Popover: 짧은 보조 정보와 필터

복잡한 업무 생성·수정은 modal이 아니라 별도 페이지를 사용한다.

## 9. 버튼 우선순위

| 유형      | 용도                  | 예시                       |
| --------- | --------------------- | -------------------------- |
| Primary   | 화면의 핵심 완료 행동 | 수주 등록, 저장, 출고 완료 |
| Secondary | 주요 행동의 대안      | 임시 저장, 미리보기        |
| Ghost     | 탐색·낮은 중요도      | 필터 초기화, 닫기          |
| Danger    | 파괴·취소·되돌리기    | 수주 취소, 사용자 비활성화 |

한 영역에 primary button을 여러 개 나열하지 않는다. 상태에 따라 가능한 행동이 여러 개면 가장 일반적인 행동 하나를 primary로 두고 나머지는 secondary 또는 menu로 제공한다.

## 10. 데이터 테이블 시스템

ERP의 핵심 공통 컴포넌트로 `DataTable`을 설계한다.

### 10.1 필수 기능

- 서버 페이지네이션
- 다중 정렬
- 컬럼별·전역 필터
- 열 표시·순서·폭 설정
- 행 선택과 허용된 bulk action
- sticky header
- 밀도 설정
- CSV/XLSX 내보내기
- URL query와 필터·페이지 상태 동기화
- 로딩, 빈 결과, 오류 상태

### 10.2 표시 원칙

- 업무번호는 상세 링크로 제공
- 금액·수량은 우측 정렬
- 날짜는 기본 `YYYY-MM-DD`
- 긴 텍스트는 말줄임 + tooltip 또는 상세 보기
- 상태는 badge와 텍스트 사용
- row action은 마지막 열에 고정
- 합계가 필터 결과 기준인지 전체 기준인지 명시
- 서버가 제공하지 않은 전체 합계를 클라이언트 현재 페이지로 계산해 전체처럼 표시하지 않음

### 10.3 선택과 대량 작업

- 현재 페이지 선택과 전체 검색 결과 선택을 구분한다.
- 대량 작업 전 대상 건수를 명확히 표시한다.
- 재고·금액·상태에 영향을 주는 대량 작업은 사유와 재확인을 받는다.

## 11. 검색과 필터

- 빠른 검색: 업무번호, 거래처, 품목명 등 대표 필드
- 상세 필터: 상태, 기간, 담당자, 부서, 품목, 창고
- 적용된 필터를 chip으로 표시
- 전체 초기화 제공
- 필터는 URL query에 반영
- 날짜 필터는 시작·종료 포함 여부를 일관되게 정의
- 검색 결과 없음과 데이터 자체 없음은 다른 empty state로 표시

## 12. 폼 설계

### 12.1 구조

```text
페이지 제목 / 상태
  → 기본 정보
  → 품목 또는 상세 행
  → 납기·담당·연결 정보
  → 첨부·메모
  → 취소 / 임시 저장 / 저장 또는 확정
```

### 12.2 검증

- 간단한 형식은 입력 중 또는 blur 시 안내
- 전체 업무 규칙은 제출 후 서버 결과를 기준으로 표시
- 오류 발생 시 첫 오류로 focus 이동
- 상단 error summary와 필드 오류를 함께 제공
- 서버 field error를 해당 입력과 연결
- 수량·금액은 표시 형식과 전송 숫자를 분리

### 12.3 저장

- 저장 중 중복 클릭 방지
- 성공 시 toast와 갱신된 상태 표시
- 실패 시 입력값 유지
- 수정된 필드가 있는 상태에서 이탈 시 경고
- version 충돌 시 덮어쓰기보다 최신값 비교 안내

## 13. 상세 화면 패턴

```text
Breadcrumb
제목 + 업무번호 + 상태 badge               주요 액션
핵심 메타: 거래처 / 담당자 / 납기 / 금액

[개요] [품목] [연결 업무] [문서] [변경 이력]
```

- 핵심 상태와 다음 행동은 tab 안에 숨기지 않는다.
- 관련 수주, 생산, 출고, AS는 링크로 연결한다.
- 변경 이력은 시간 역순으로 표시하고 변경 필드·이전값·변경값·사유를 제공한다.
- 중요 금액과 수량은 단위와 VAT 포함 여부를 명시한다.

## 14. 업무 상태 표현

상태 색상은 도메인 전체에서 의미를 일관되게 사용한다.

| 의미           | 표현                              |
| -------------- | --------------------------------- |
| 초안·미시작    | neutral                           |
| 요청·대기      | info                              |
| 진행 중        | primary                           |
| 주의·지연·부족 | warning                           |
| 완료·정상      | success                           |
| 취소·실패·차단 | danger 또는 neutral + 명확한 라벨 |

상태 라벨은 API code를 그대로 노출하지 않고 한국어 display label로 매핑한다.

## 15. 피드백 상태

모든 데이터 화면은 다음 상태를 설계한다.

### Loading

- 초기 목록·상세: skeleton
- 버튼 작업: 버튼 내부 spinner + 중복 클릭 방지
- 장시간 작업: 진행 상태, 작업 ID, 화면 이탈 가능 여부 표시

### Empty

- 데이터 없음: 생성 권한이 있으면 첫 행동 안내
- 필터 결과 없음: 필터 해제 또는 수정 안내
- 권한 범위 내 데이터 없음: 권한 오류로 오해하지 않게 설명

### Error

- 사용자 메시지, 재시도, 필요 시 문의용 correlation ID
- 입력 오류는 필드 가까이 표시
- 전체 페이지 오류와 부분 영역 오류를 구분
- 기술 stack trace를 사용자에게 표시하지 않음

### Success

- 짧은 작업은 toast
- 상태가 중요한 작업은 화면 본문에도 결과 반영
- 비동기 작업은 작업 센터 또는 알림에서 결과 확인

## 16. 확인과 위험 작업

확인 dialog는 다음 작업에 사용한다.

- 수주·발주·출고·AS 취소
- 재고 조정
- 사용자 비활성화
- 확정 이후 금액·수량·납기 변경
- 대량 변경·내보내기

dialog에 포함할 내용:

- 대상 업무번호와 이름
- 발생하는 결과
- 되돌릴 수 있는지 여부
- 필요한 변경 사유
- 명확한 동사형 확인 버튼

`확인하시겠습니까?`만 표시하는 일반 문구는 사용하지 않는다.

## 17. 대시보드

- 역할별로 필요한 KPI를 우선 표시
- 숫자만 나열하지 않고 기간과 비교 기준 제공
- KPI card 클릭 시 해당 필터가 적용된 목록으로 이동
- 지연, 부족, 실패, 승인 대기 등 행동 가능한 예외를 우선
- chart에는 표 또는 accessible summary 제공
- 실시간이 아닌 데이터는 최종 갱신 시각 표시

### 17.1 홈 구성

```text
환영 문구 / 오늘 날짜
내 할 일: 승인 대기, 생산 요청, 출고 준비, AS 배정
주요 예외: 납기 지연, 재고 부족, 이관 실패
최근 활동
즐겨찾기 또는 빠른 메뉴
```

## 18. 내비게이션

- 현재 위치는 sidebar와 breadcrumb에서 모두 식별 가능해야 한다.
- 메뉴명은 부서가 아닌 업무 대상을 우선한다.
- 브라우저 back/forward가 정상 동작해야 한다.
- 목록에서 상세로 갔다가 돌아오면 필터, 정렬, 페이지를 복원한다.
- 관리자 메뉴는 일반 사용자 흐름과 분리한다.
- 전역 검색은 2차 기능으로 업무번호, 고객, 제품을 대상으로 설계한다.

## 19. 권한 UI

- 읽기 권한 없음: 메뉴 숨김, URL 접근 시 권한 없음 화면
- 읽기 가능·수정 불가: 값과 이력은 표시하되 편집 action 숨김 또는 명확한 disabled 사유
- 상태상 불가: 권한과 구분해 “현재 상태에서는 출고할 수 없음” 표시
- 승인 필요: 요청 행동과 승인 행동을 분리
- 관리자는 다른 역할로 보이는 UI를 테스트할 수 있는 기능을 별도 검토

권한이 없는 이유나 상태상 불가능한 이유가 사용자에게 도움이 되는 경우 tooltip 또는 안내문을 제공한다.

## 20. 문서 변환 UI

```text
1. PDF 업로드
2. 추출 진행
3. 추출 결과와 원문 비교
4. 필드·품목 수정
5. 사용자 확정
6. XLSM 생성 및 이력 저장
```

- 자동 추출값과 사용자가 수정한 값을 구분 가능하게 한다.
- confidence가 낮거나 필수값이 없는 필드를 강조한다.
- 생성 전 템플릿 버전과 주요 값을 확인한다.
- 실패 시 원본을 다시 올리게 하지 않고 가능한 재시도 경로를 제공한다.

## 21. 이관·동기화 UI

- 작업 상태: 대기, 처리 중, 성공, 일부 성공, 실패
- 원본 파일, 시트, 시작자, 시작·종료 시각 표시
- 전체·성공·경고·실패 행 수 표시
- 오류 파일 다운로드
- 재시도와 취소 가능 여부 표시
- 운영 반영 전 대사 결과와 승인 단계 제공
- 작업 실행 버튼은 관리자 permission으로 제한

## 22. 접근성

- 모든 기능은 키보드로 조작 가능해야 한다.
- focus indicator를 제거하지 않는다.
- icon-only button은 accessible name을 제공한다.
- 입력 label, error, help text를 연결한다.
- modal 열림 시 focus 이동, 닫힘 시 원래 위치 복귀
- table header와 정렬 상태를 보조기술에 전달
- 색 대비 WCAG AA 목표
- 오류·성공을 색상만으로 표시하지 않음
- 애니메이션 감소 설정 존중

## 23. 반응형 기준

| 구간    |       기준 | 동작                                 |
| ------- | ---------: | ------------------------------------ |
| mobile  |    < 768px | drawer nav, 단일 컬럼, 핵심 열/카드  |
| tablet  | 768~1199px | 축소 sidebar, 2열 폼 가능            |
| desktop |   ≥ 1200px | 고정 sidebar, 넓은 표, 다열 레이아웃 |

주 사용 환경은 desktop Chrome/Edge다. 모바일에서 지원하지 않는 대량 편집은 읽기 전용과 명확한 안내를 제공한다.

## 24. 국제화와 형식

- 기본 언어: 한국어
- 기본 timezone: Asia/Seoul
- 날짜: `YYYY-MM-DD`, 필요 시 시각 `YYYY-MM-DD HH:mm`
- 통화: `₩` 또는 `원`, 천 단위 구분
- VAT 포함·별도 여부를 금액 라벨에 명시
- 수량에는 단위 표시
- API code와 사용자 표시 라벨을 분리해 향후 다국어 확장 가능하게 구성

## 25. 프론트엔드 구현 규칙

- TypeScript strict mode 사용
- OpenAPI 기반 생성 타입을 우선 사용
- `any`로 API 타입 오류를 회피하지 않음
- React Query key를 feature 단위로 표준화
- mutation 성공 후 관련 query invalidation 명시
- 공통 UI에 업무 로직을 넣지 않음
- 폼 schema는 Zod로 정의하고 서버 검증을 대체하지 않음
- 날짜·금액 format helper를 공통화
- feature flag가 꺼진 기능의 경로와 action을 노출하지 않음

## 26. 컴포넌트 상태 검증

공통 컴포넌트는 Storybook 또는 동등한 환경에서 다음 상태를 확인한다.

- hover/focus/active
- disabled
- loading
- empty
- error
- 긴 한국어 텍스트
- 매우 큰 금액·수량
- 좁은 화면
- 키보드 탐색
